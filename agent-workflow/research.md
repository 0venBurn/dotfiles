# Directing AI coding agents: the shell harness playbook in 2026

The discipline of "context engineering" — structuring the information that flows into an AI coding agent's context window — has rapidly emerged as the decisive factor in agent effectiveness, eclipsing prompt engineering in importance. Across Claude Code, Cursor, OpenAI Codex CLI, Aider, Devin, Windsurf, GitHub Copilot, and Continue.dev, a shared architecture is converging: persistent markdown instruction files loaded at session start, slash commands defined as markdown with YAML frontmatter, and multi-layered context budgeting systems that compress ruthlessly to keep agents on track. This report synthesizes the current state of practice across all three axes — persistent context files, slash command design, and context injection — drawing on official documentation, research papers, leaked internals, and community consensus.

## Persistent context files are converging on a single standard

The most consequential development is the emergence of **AGENTS.md as a cross-tool standard**, donated to the Agentic AI Foundation under the Linux Foundation in December 2025 alongside Anthropic's MCP and Block's Goose. AGENTS.md is now supported by OpenAI Codex, GitHub Copilot, Cursor, Windsurf, Google Jules/Gemini, and over 60,000 repositories. Meanwhile, each tool retains its native format — `CLAUDE.md` for Claude Code, `.cursor/rules/*.mdc` for Cursor, `.github/copilot-instructions.md` for Copilot, `.windsurf/rules/*.md` for Windsurf, `CONVENTIONS.md` for Aider, `.continue/rules/*.md` for Continue.dev — but cross-reading AGENTS.md has become table stakes.

The file hierarchy matters more than the filename. Claude Code walks from the current working directory up to the filesystem root, loading `CLAUDE.md` and `CLAUDE.local.md` at each level, with more specific (closer to CWD) files taking priority. OpenAI Codex does the same with `AGENTS.md`, concatenating from root down so files closer to CWD appear later in the prompt and override earlier guidance. Cursor, Copilot, and Windsurf all support glob-scoped rules via YAML frontmatter that activate only when matching files are in context. This conditional loading pattern — where a rule about TypeScript testing only activates when `.test.ts` files are touched — represents the sharpest divergence from the original monolithic approach.

Content-wise, strong consensus has formed around what belongs in these files and what doesn't. Anthropic's official guidance says to include bash commands the agent can't guess (build, test, lint), code style rules that differ from language defaults, testing instructions, repository etiquette, and common gotchas. Equally important: exclude anything the agent can figure out by reading code, standard language conventions, detailed API docs (link instead), information that changes frequently, and self-evident practices like "write clean code." The community framework from HumanLayer recommends organizing around **WHAT** (tech stack, project structure), **WHY** (purpose and function), and **HOW** (workflow, verification commands). GitHub's recommended sections for Copilot are project overview, tech stack and tooling, build/test/validation commands, coding standards, and architecture.

The critical finding on file length comes from multiple converging sources. Professional teams keep their context files **under 200 lines**, with many successful examples under 60 lines. There is roughly a 150–200 instruction budget before compliance drops off, and the system prompt already consumes approximately 50 of those slots. A Princeton study by Lulla et al. (January 2026) across 10 repositories and 124 pull requests found that well-crafted AGENTS.md files produced a **28.6% median runtime reduction** and 16.6% fewer output tokens. However, an ETH Zurich study from February 2026 delivered a sobering counterpoint: auto-generated context files actually reduced task success rates by roughly 3% and increased costs by over 20% compared to having no context file at all. Hand-written, concise files win decisively.

## SKILL.md introduces progressive disclosure for on-demand knowledge

The most architecturally significant pattern is **SKILL.md**, now an open standard at agentskills.io, supported by Claude Code, OpenAI Codex CLI, GitHub Copilot, Gemini CLI, and Cursor. Skills solve the fundamental tension between comprehensive context and context window budget through three-level progressive disclosure. At **Level 1**, only the skill's name and description (~100 tokens) are loaded at startup. At **Level 2**, the full SKILL.md instruction body loads when the agent determines a task matches the description. At **Level 3**, scripts, templates, and reference documents load on demand during execution. A skill directory follows this structure:

```
.claude/skills/my-skill/
├── SKILL.md         # Required: instructions + metadata
├── scripts/         # Optional: executable code
├── references/      # Optional: documentation
└── assets/          # Optional: templates, resources
```

This pattern represents a fundamental shift from "front-load everything" to "discover as needed." Cursor's engineering blog (January 2026) reported that moving agent skills from always-loaded system prompts to discoverable files — where the agent uses grep or semantic search to load them on demand — was one of five techniques that contributed to a **46.9% reduction in total agent tokens** during MCP-tool-calling runs. The lesson is clear: the best context strategy is not about maximizing what's in the window, but minimizing what's in the window while ensuring everything needed is reachable.

## Slash commands are splitting into three tiers

Across all major tools, slash command interfaces have settled into a three-tier architecture. **Built-in commands** handle session management and are hardcoded: `/clear`, `/compact`, `/model`, `/cost`, `/help`, `/exit`. **User-defined commands** (increasingly called "skills") handle project-specific workflows and are defined as markdown files in conventional directories: `.claude/skills/`, `.cursor/commands/`, `~/.codex/prompts/`. **Integration commands** are auto-generated from connected services, most prominently via MCP: `/mcp__github__list_prs`, `/mcp__jira__create_issue`.

The naming taxonomy debate has produced three distinct schools. **Action-oriented** commands (`/fix`, `/explain`, `/refactor`, `/commit`) dominate GitHub Copilot, Aider, and Continue.dev — they're intuitive and map directly to developer intent. **Resource-oriented** commands (`/add`, `/drop`, `/model`, `/memory`) dominate Aider and parts of Claude Code, requiring users to understand the system's architecture. **Modal commands** (`/code`, `/ask`, `/architect`, `/plan`, `/bypass`) are a distinctive feature of Aider and Devin, changing the agent's persistent behavior rather than triggering a one-shot action.

The most instructive community example is the wshobson/commands repository, which provides 57 production-ready Claude Code commands explicitly organized into **15 workflows** (`/workflows:feature-development`, `/workflows:smart-fix`) and **42 tools** (`/tools:api-scaffold`, `/tools:security-scan`). This encodes the coarse vs. fine granularity distinction directly into directory structure, suggesting the answer to the granularity debate is "both, explicitly separated." Workflow commands orchestrate multi-step processes; tool commands do one thing precisely.

Context flows into commands through multiple mechanisms, and the differences matter. Claude Code's custom commands support **dynamic shell expansion** — backtick-wrapped commands like `` !`git status` `` execute at invocation time and inject live output into the prompt. OpenAI Codex supports **positional arguments** (`$1` through `$9`) and **named parameters** (`FILES=src/ PR_TITLE="title"`). Cursor commands operate on currently selected code and open files implicitly. Continue.dev uses the `{{{ input }}}` template variable. GitHub Copilot's unique three-pronged `@participant /command #context` addressing scheme — where `@workspace /fix #file:auth.ts` combines an AI participant, an action, and explicit context — is the most structured approach.

On composability, no tool supports explicit pipe-style chaining (`/lint | /fix | /commit`). The closest patterns are Claude Code's `/batch` skill, which decomposes work into 5–30 units and spawns parallel background agents in isolated git worktrees; Windsurf's numbered workflow convention (`/0-task` → `/1-discovery` → `/2-design` → `/3-implement` → `/4-clean`); and external shell scripting around CLI tools. Claude Code's `/loop` skill adds temporal composition, running prompts on a configurable schedule. Session forking (`/fork` in Claude Code, Codex, and Devin) enables branching a conversation to explore alternatives without polluting the main context.

## Context injection follows a universal six-layer architecture

Despite radically different implementations, every major coding agent harness has converged on a common six-layer context architecture: a system prompt layer (persistent instructions plus tool definitions), a codebase awareness layer (AST parsing plus embeddings plus graph ranking), a dynamic retrieval layer (agent-initiated file reads and searches), a conversation history layer (rolling window with compression), an external memory layer (scratchpads and cross-session memories), and a context budget manager (tiered compression from snipping to nuclear compaction).

**Aider's repository map** is the most transparent implementation of codebase awareness. It uses tree-sitter to parse source files into definition and reference tags, builds a directed graph where nodes are files and edges represent code references, then applies **PageRank with personalization** based on currently active chat files to rank file importance. A binary search fits the most important content within a configurable token budget (default 1,024 tokens, expanding to 8,192 when no files are selected). The output is a hierarchical tree of file paths and their key definitions — giving the agent structural awareness without reading entire files.

**Windsurf's Cascade** takes the most aggressive ambient approach, indexing the entire local codebase with 768-dimensional vector embeddings and tracking real-time actions (file edits, terminal commands, clipboard activity, navigation) for dynamic context assembly. **Sourcegraph Cody** uses a hybrid dense-sparse vector retrieval system published at RecSys '24, combining keyword search with semantic search via an Expand and Refine method on a repo-level semantic graph. **Cursor** has moved furthest toward dynamic context discovery, preferring to let the agent pull context on demand rather than statically pre-loading it.

## How the context window budget actually works

The most revealing technical detail comes from Claude Code's four-layer compaction pipeline, which executes at the start of every agent iteration within a ~200K token window (or 1M for Max/Team/Enterprise plans). The philosophy is to **avoid calling an LLM for summarization unless absolutely necessary**. Layer 1 ("Snip") simply drops entire older messages. Layer 2 ("Microcompact") replaces the content of recomputable tool results — file reads, shell output, grep results, web fetches — with `'[Old tool result content cleared]'`, preserving the tool call structure so the agent knows what it already looked at. Layer 3 ("Context Collapse") groups messages into segments and uses an LLM to summarize older ones, triggered at **90% context utilization** (non-blocking) or 95% (blocking). Layer 4 ("Autocompact") is the nuclear option: summarizing the entire conversation, then re-injecting critical state including open files, agent listings, MCP instructions, and invoked skills.

A typical token breakdown for a Claude Code session reveals the budget allocation: system prompt consumes about **4% of the window** (~8,200 tokens, split between tool schemas, CLAUDE.md content, and agent instructions), conversation history takes ~46%, tool results take ~27%, and approximately 10% is reserved as a response buffer.

GitHub Copilot takes a fundamentally different approach: it **rebuilds the entire prompt from scratch on every iteration** rather than mutating the history incrementally. Each message and tool call gets a priority score and a `flexGrow` weight (modeled after CSS flexbox), with newer turns receiving proportionally more budget. Results exceeding 8KB are replaced with pointers (except for search and execution subagent results). When budget is still exceeded, Copilot selects exactly one of four summarization strategies: foreground summarization, inline summarization (with a 15% budget expansion), background compaction, or delegation to OpenAI's Responses API for server-side compaction.

The most advanced research on this front is the **CAT (Context as a Tool)** framework from December 2025, which proposes elevating context management to a callable tool in the agent's decision loop. It structures the context workspace into three tiers: stable task-semantic anchors (goals and constraints that persist always), evolvable long-term memory (condensed summaries created at milestones), and short-term working memory (high-fidelity recent interactions). The agent proactively triggers context folding at stage boundaries. The associated SWE-Compressor model achieved **57.6% on SWE-Bench-Verified**, significantly outperforming ReAct-based agents and static compression baselines.

## Structured formatting has measurable but model-specific effects

Research on whether XML, JSON, markdown, or plain text affects agent performance shows significant impact — but the optimal format depends on the model. A study on prompt formatting (arXiv 2411.10541) found GPT-3.5-turbo's performance varied by up to **40%** depending on template format, with GPT-3.5 preferring JSON while GPT-4 favored markdown. Cross-model transferability of optimal formats was extremely low (IoU often below 0.2), though models from the same family showed high agreement.

In practice, **markdown is the most token-efficient format** — roughly 10% fewer tokens than YAML for equivalent content and dramatically fewer than XML, which uses approximately **80% more tokens** than markdown for the same data. However, Anthropic's models are specifically tuned to pay attention to XML tag structure, and Claude Code internally uses XML-style tags for system warnings and budget tracking (`<system_warning>Token usage: 35000/1000000</system_warning>`). The pragmatic consensus: use markdown as the primary format for rules, instructions, and documentation context, but use XML tags for structural delineation of distinct context sections when targeting Claude models. As one cross-vendor analysis noted, XML tags are the only format that all models from Anthropic, Google, and OpenAI explicitly encourage for prompt structuring.

Stanford research by Liu et al. found that LLMs perform **over 30% worse** when relevant information is buried in the middle of the context window compared to the beginning or end. This "lost in the middle" effect has direct implications for context file design: the most critical instructions should appear at the top of CLAUDE.md/AGENTS.md files, and compaction strategies should preserve the beginning and end of conversation histories.

## Conclusion: what the convergence reveals

Five concrete recommendations emerge from the current state of practice. First, keep your primary context file under 200 lines, hand-written, and organized as WHAT/WHY/HOW — never auto-generate it. Second, use SKILL.md for domain-specific knowledge that should load on demand rather than consuming context budget at startup. Third, structure custom commands in two explicit tiers — workflows for multi-step orchestration and tools for single-purpose actions — using directory prefixes to distinguish them. Fourth, prefer dynamic context discovery (agent-pulled) over static context injection (always-loaded) for anything beyond core project conventions. Fifth, use markdown as your default format but wrap structurally distinct sections in XML tags when targeting Anthropic models.

The deepest open question is whether harness-level context management will be superseded by model-native context awareness. Cognition's testing with Claude Sonnet 4.5 found the model was already context-window-aware — burning through parallel tool calls faster early in the window and becoming cautious near the limit. Yet their existing memory systems still outperformed the model's own awareness "when we explicitly directed the agent to use its previously generated state." For now, the harness remains essential. The tools that win will be those that make context engineering invisible — compressing aggressively, loading progressively, and keeping the agent's attention focused on what matters for the current task rather than drowning it in everything it might someday need.
